# Java 練習問題：BankAccount と BankService

## 概要
銀行の口座情報と、複数の口座を管理・操作するサービスプログラムを作成します。
カプセル化（データ保護）と、クラスの責務（例外を投げるべきか、戻り値で失敗を伝えるべきか）を意識して設計・実装してください。

**【制約事項】**
Stream API、`List`（`ArrayList`など）、`record`、Lombok は使用せず、**配列と通常の制御構文（for文、if文など）のみ**を使用して実装すること。

---

## 1. BankAccount クラスの仕様
銀行口座1つ分の情報を管理するクラスです。不正なデータが絶対に入らないよう、堅牢な設計にしてください。

### フィールド
* `id` (整数): 口座ID
* `ownerName` (文字列): 名義人
* `balance` (整数): 残高

### コンストラクタ
* **`BankAccount(int id, String ownerName, int balance)`**
  引数の値を各フィールドに設定します。ただし、銀行のシステムとして以下の「契約違反（不正なデータ）」が渡された場合は、オブジェクトの生成を拒否し **`IllegalArgumentException`** を投げてください。
  * `id` が `0` 以下の場合
  * `ownerName` が `null`、空文字（`""`）、または空白のみ（`" "`など）の場合
  * `balance` が `0` 未満（マイナス）の場合

### メソッド
* **`getId()`** / **`getOwnerName()`** / **`getBalance()`**
  * 各フィールドの値を返す。
* **`setOwnerName(String ownerName)`**
  * 名義人を更新する。コンストラクタと同じルールで不正な文字列を弾くこと。
* **`deposit(int amount)`**
  * 口座に入金する。
  * `amount` が `0` 以下の場合は、操作として不正であるため `IllegalArgumentException` を投げる。
* **`withdraw(int amount)`**
  * 口座から出金する。
  * `amount` が `0` 以下の場合は、操作として不正であるため `IllegalArgumentException` を投げる。
  * **【業務要件】** 金額指定自体は正しいが「残高不足」で出金できない場合は、プログラムの異常ではなく「業務上普通に起こりうる失敗」として扱うため、例外は投げずに **`false`** を返す。成功した場合は残高を減らして **`true`** を返す。
* **`printInfo()`**
  * 以下の形式で標準出力に表示する。
  * 出力例: `[ID: 101] Name: Taro, Balance: 5000`

## 2. BankService クラスの仕様
複数の口座をまとめて管理し、送金などの操作を行うサービスクラスです。

### フィールド
* `accounts` (BankAccountの配列)

### コンストラクタ
* **`BankService(BankAccount[] accounts)`**
  引数の配列を元に初期化します。以下のルールを守ってください。
  * 引数が `null` の場合は、長さ0の配列として扱う。
  * 配列内の `null` 要素は除去し、有効な口座データだけを隙間なく詰めた新しい配列を作成して保持する。

> **【データ保護要件】**
> インスタンス生成後、引数として渡された大元の配列や、大元の `BankAccount` オブジェクトそのものが外部で変更されても、`BankService` 内部の口座データが一切影響を受けないように実装してください。

### メソッド
* **`findById(int id)`**
  * 指定した `id` の口座オブジェクトを返す。見つからなければ `null` を返す。
  * **【データ保護要件】** 戻り値のオブジェクト経由で、勝手に内部データが操作されないようにすること。
* **`transfer(int fromId, int toId, int amount)`**
  * `fromId` の口座から `toId` の口座へ送金する。
  * 以下のいずれかの場合は失敗として `false` を返す。（※ `amount` が `0` 以下の場合は例外ではなく `false` とすること）
    * `amount` が `0` 以下の場合
    * 送金元、または送金先が見つからない場合
    * 送金元の残高が不足している場合
  * 成功したら、送金元から引き出し、送金先へ入金して `true` を返す。
  * **【実装のポイント】** ここでの操作結果は、`BankService` が管理している**内部の本物の口座データ**に反映されなければなりません。
* **`totalBalance()`**
  * 管理している全口座の残高の合計値を返す。
* **`printAll()`**
  * 全口座の `printInfo()` を順番に呼び出す。

## 3. 動作確認用コード（Main）
作成したクラスが正しく仕様を満たしているか確認するため、`Main` クラスを作成し、以下の動作を検証するテストコードを記述してください。

**【テスト必須項目】**
1. **正常系のテスト**:
   * 口座を2つ作成し、`BankService` に登録する。
   * 片方の口座からもう片方の口座へ、正常に送金処理（`transfer`）が行えることを確認する。
2. **業務上の失敗（false）のテスト**:
   * 残高を超える金額を送金しようとした際、送金処理が失敗（`false` が返る）し、両者の残高が変わっていないことを確認する。
3. **契約違反（例外）のテスト**:
   * `try-catch` 文を使用し、ある口座に対して「マイナスの金額（例：`-100`）」を入金（`deposit`）しようとした際に、想定通り `IllegalArgumentException` が発生し、エラーメッセージが出力されることを確認する。
4. **データ保護（カプセル化）のテスト**:
   * `findById` で口座情報を取得し、その取得したオブジェクトの残高を不正に書き換える（例：直接 `withdraw` を実行するなど）。
   * その後、`BankService` 側の `totalBalance()` または `printAll()` を呼び出し、内部のデータが一切壊れていない（カプセル化が守られている）ことを証明する。

## ヒント

<details>
<summary>1. 「例外」と「false」の使い分けについて</summary>

* **例外（IllegalArgumentException）**: プログラムの書き間違いや、メソッドの前提条件を破るような「契約違反」の操作に対して投げます（例：マイナスの金額を入金しようとするなど）。
* **falseで返す**: 操作の形式自体は正しいが、たまたま条件が合わずに処理できなかった「業務上の失敗」に対して使います（例：正しい金額を出金しようとしたが、たまたま残高が足りなかったなど）。

</details>

<details>
<summary>2. データ保護（カプセル化）の基本</summary>

フィールドを `private` にし、配列やオブジェクトを外部から受け取ったり返したりする際は、そのまま渡さずに `new` して複製する（コピーコンストラクタ等を使う）のが基本です。

</details>

<details>
<summary>3. transfer（送金）メソッドの罠</summary>

送金処理では、コピーデータではなく **「`BankService` 内部に保存されている本物のデータ」** の残高を更新する必要があります。

もし、送金元の口座を探す際に `findById` を使ってしまうと、`findById` は「データ保護のためにコピーを返す」仕様であるため、いくら残高を減らしても本物のデータは更新されません。
外部公開用の `findById` とは別に、**内部操作専用の検索メソッド（本物の参照を返す `private` メソッド）** を作ると綺麗に解決します。

</details>
